# Chrome extensions

Portable, project-agnostic practices for building Chrome (and Chromium) extensions — manifest versions, background/service workers, content scripts, permissions, packaging and store review — true for any extension read cold.

> **Releasing, versioning, and Chrome Web Store publication are standardized** — before building or changing any release/publish machinery in an extension repo, read the [chrome-extension-release standard](../chrome-extension-release/RELEASE.md) (its pack, declared when you're ready to ship) and copy its canonical workflows instead of re-deriving them.

- An MV3 service worker's relative paths resolve against the worker's **own** file location, not the extension root. Any path handed to a Chrome API (`importScripts`, `action.setIcon`) or `fetch` must be extension-root absolute — a leading slash, or `chrome.runtime.getURL(...)`. A bare relative path silently fails: it can abort the worker on import, or make an API call reject with a fetch error while the worker keeps running.
- `chrome.declarativeContent`'s `SetIcon` action needs `imageData`, not `path` — the documented `path` option is unreliable in practice (it can silently leave the icon unset). `declarativeContent` rules are evaluated by the browser process, so the icon must already be raw pixels at registration time; since a service worker has no DOM, build the `imageData` by decoding the packaged icon yourself: `fetch(getURL(icon)) → blob → createImageBitmap → OffscreenCanvas.drawImage → getImageData`.
- `chrome.events.UrlFilter`'s host operators are **raw string matches, not domain-boundary matches** — `hostSuffix: "example.com"` also matches `evilexample.com`, and `hostContains: "example"` matches any host containing that substring. `declarativeContent`'s `PageStateMatcher.pageUrl` gates an action icon or page condition on exactly these filters, so a **lookalike host silently satisfies a rule meant for your own site**. To mean *apex-or-any-subdomain*, combine `hostEquals: "example.com"` with `hostSuffix: ".example.com"` (the leading dot forces a label boundary); never gate a security- or origin-sensitive behavior on a bare `hostSuffix`.
- A shared global assembled by multiple injected content-script files, re-injected into the page on every activation (e.g. every popup open), must be **augmented**, never replaced — merge onto it (e.g. `Object.assign`), not `globalThis.X = {...}`, or each newly-injected file wipes what earlier files already attached. Likewise reset any state a file accumulates on each injection (e.g. a list a source pushes into) at load time, or it keeps growing across re-injections instead of staying per-load.
- Introspecting an MV3 service worker over CDP (`Runtime.evaluate` inside it) has two traps: `chrome.*` callback APIs don't reliably settle when awaited with `awaitPromise: true` (a hang with no internal timeout) — build the awaited signal from plain promises (`fetch`/`OffscreenCanvas`) instead of a `chrome.*` callback; and a bare top-level `function`/`const` isn't reachable from an injected evaluate — expose what the probe reads as an explicit `globalThis.x = …`. A dormant worker also has no globals until it re-runs its top level, and attaching to it is what starts that — so poll for the global rather than reading it once immediately after attaching.
- To authenticate an extension to a **JWT-validating** backend (an API Gateway JWT authorizer, any OIDC-validating server), obtain a Google **ID token** with `chrome.identity.launchWebAuthFlow` (`response_type=id_token`) against a Google Cloud OAuth client of type **Web application**, redirect to `https://<extension-id>.chromiumapp.org/` (from `chrome.identity.getRedirectURL()`), scope `openid email profile`, and **verify the returned `nonce`**. Do **not** use `chrome.identity.getAuthToken` — it returns an opaque OAuth *access* token (no verifiable signature/`iss`/`aud`) that a JWT authorizer rejects. Pin the extension id with a manifest `key` so the redirect URI stays fixed.
- MV3 loads ES modules natively — don't add a bundler just to use `import`/`export`. Declare the service worker `"type": "module"` and load page/side-panel scripts with `<script type="module">`; relative imports resolve within the packaged extension.
- A silent token refresh (`launchWebAuthFlow({interactive:false})`) can only complete a flow that needs no UI, so it must request `prompt=none`; `prompt=consent` always needs interaction and therefore always fails silently — reserve it (or omit `prompt`) for the interactive fallback.
- On a **silent** token refresh with more than one account signed in, pass the remembered account's email as `login_hint` — otherwise the provider can't tell which session to reuse and forces an interactive account-picker, so the silent (`prompt=none`) refresh fails. Persist the non-secret email from the first successful sign-in for exactly this purpose.
- Extension storage is unencrypted, so treat tokens as secrets at rest: keep the bearer/ID token in in-memory `chrome.storage.session` (cleared on browser exit) and persist only non-secret identifiers (e.g. the account email) to `chrome.storage.local`. Don't switch to a refresh-token flow just to survive restarts — that puts a longer-lived credential on disk; re-run the silent flow instead.
- To run a content script on arbitrary third-party pages **without** an install-time host warning, request access at runtime: declare the origins under `optional_host_permissions` plus the (silent) `scripting` permission, call `chrome.permissions.request()` **synchronously inside a real foreground user gesture** (Chrome rejects the request from a service-worker message handler), then register the script dynamically with `chrome.scripting.registerContentScripts()` — never a static `content_scripts` entry. The grant can be revoked from `chrome://extensions` out from under you, so on every service-worker start reconcile your stored enabled-flag against the permission actually granted and re-register (or clean up) to match.
- MV3 gives the side panel no is-open/close API, and an action popup suppresses `action.onClicked` — so to know whether your panel is open, have the panel open a `Port` on load and read its connect/disconnect as open/closed. Opening the panel programmatically needs `chrome.sidePanel.open()` (Chrome 116+), so raise `minimum_chrome_version` accordingly.
- `host_permissions` does not bypass CORS. Listing a host lets the extension's fetches *reach* it, but the server must still return CORS headers for the extension origin (`chrome-extension://<id>`) or the request fails in-browser despite correct client code. Conversely, don't add a `host_permissions` entry merely to reach *your own* backend: if it already returns permissive CORS the fetch succeeds without it, and the entry only adds a scary install-time host-access warning for no functional gain.
- A **registered or static content script runs as a *classic* script — it can't be an ES module.** `chrome.scripting.registerContentScripts` and static `content_scripts` inject their files as classic scripts (there is no `type: 'module'` mode, unlike a page's `<script type="module">` or the module service worker), so a top-level `import` in one throws `Uncaught SyntaxError: Cannot use import statement outside a module` — **in the host page's console, not the extension's** — and the feature silently no-ops while the extension console stays clean. To load module code without a bundler, register a tiny **classic loader** whose only statement is a *dynamic* `import(chrome.runtime.getURL('…'))`: dynamic `import()` is legal in a classic script, and the module it pulls in runs in the same content-script isolated world with the content-script `chrome.*` surface intact. List that module **and its whole transitive import graph** under `web_accessible_resources` (gated to the target origins) or the fetch is blocked — and have a test walk the graph from the entry module so a newly-added import can't silently fall out of the list.
- To put a menu on the **toolbar icon itself**, create it with `chrome.contextMenus.create({ contexts: ['action'], … })` — the `contextMenus` permission carries **no** install-time warning (unlike a broad host permission). Recreate the items with `contextMenus.removeAll()` first on install/startup so they self-heal instead of throwing on duplicate ids when the service worker re-runs its top level.
